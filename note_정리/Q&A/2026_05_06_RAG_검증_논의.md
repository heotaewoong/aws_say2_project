## Q2. Faithfulness가 뭔가?

**A:** RAG 평가의 표준 지표로 **"AI 답변이 제공된 자료에 근거했는가"** 를 0~1로 측정.

```
[AI에게 준 자료 (context)]
PubMed: "Sirolimus는 LAM 환자의 FEV1 감소를 유의하게 억제함"

[AI가 쓴 소견서]
답변 ①: "Sirolimus가 FEV1 저하를 막음"     ← context 근거 ✅
답변 ②: "Sirolimus는 부작용으로 구토 유발"  ← context 밖 내용 ❌ (환각)

Faithfulness = 근거 있는 주장 수 / 전체 주장 수
답변 ①: 1/1 = 1.0 (완벽)
답변 ②: 0/1 = 0.0 (환각)
```

우리 시스템 0.62 = "Top 1 설명은 근거 기반인데, Top 2/3 감별진단 설명할 때 가끔 PubMed에 없는 일반 의학 지식을 섞어 씀."

## Q3. 실제 병변과 예측이 맞는지 확인을 해야 하는 거 아닌가?

**A:** 네 말이 맞다. 기존 검증 보고서의 검증은 진짜 검증이 아니라는 걸 인정해야 함.

| 레벨 | 실제 한 일 | 문제점 |
|------|-----------|--------|
| Level 1 | 4,293개 질환 각각에 그 질환의 HPO 3개를 넣어 자기 자신이 나오는지 확인 | **자기 자신을 넣고 자기 자신을 찾음** — 실제 환자 검증 아님 |
| Level 2 | 교과서에서 LAM/IPF 등 5개 질환을 보고 직접 증상 텍스트 작성 | **답을 보고 시험지 만든 것** |
| Level 3 | MIMIC 실환자 1명 돌렸지만 간경화 환자 | **실제 폐질환 환자 검증 없음** |

진짜 검증에 필요한 것:
1. 실제 폐질환 환자 데이터 확보
2. 실제 진단 (Ground Truth) 확보
3. 파이프라인 실행 → Top 3 출력
4. Top 3 안에 실제 진단 들어오는지 확인

## Q4. CheXpert 데이터로 검증하는 게 맞지 않나?

**A:** 절반만 맞다. 시스템은 두 레이어로 구분해서 검증해야 함.

### 레이어 1 — SooNet (X-ray → 14개 병변 분류)

| | |
|---|---|
| 입력 | X-ray 이미지만 |
| 출력 | 14개 병변 확률 (Atelectasis, Cardiomegaly 등) |
| 검증 데이터 | **CheXpert valid 234장 (S3: `cheXpert_data/valid_only/`)** |
| 검증 지표 | AUROC, F1, Sensitivity, Specificity |
| 이미 있는 코드 | `scripts/eval/eval_soonet_local.py` |
| 이게 증명하는 것 | **"실제 병변 vs 예측 병변"** 일치도 |

### 레이어 2 — RAG 파이프라인 전체

| | |
|---|---|
| 입력 | X-ray + 증상 텍스트 + Lab + 기본정보 |
| 출력 | Top 3 희귀질환 + JSON 소견서 |
| 검증 데이터 | **CheXpert로는 불가능** (증상/Lab/진단 없음) |
| 필요한 데이터 | MIMIC-CXR + MIMIC-IV discharge + labevents |
| 검증 지표 | Top-3 Hit Rate, PMID 환각율 |

**CheXpert 데이터셋에 없는 것**:
- 증상 텍스트(의사 소견서) ❌
- 혈액검사 수치 ❌
- 환자 최종 진단명(ICD-10) ❌
- 희귀질환 정보 ❌

→ CheXpert로는 **SooNet만** 검증 가능, **RAG 전체는 MIMIC 필요**.

- 결과: Hit@1, Hit@3, PMID 유효율

두 결과를 `RAG_통합_검증보고서.md`에 통합 반영.

## Q5. 그런데 우리 회의록에서 주요질환 랭킹, 희귀질환 listing(근거 기반 질환 제시, 유전자 검사 권고, 치료 가이드라인, 최신 치료 동향) 이런 식으로 뽑기로 하지 않았나?

**A:** 맞다. 2차 회의록 (`2차_RAG회의록_20260421 (1).docx`) §1.4에서 확정된 출력 구조가 바로 그것임.

```
주요 질환 → 랭킹 형태 출력 (Top 10)
희귀질환 → 유전자 리스트 + 아래 4가지:
   ① 근거 기반 질환 제시 (케이스리포트 + Orphadata)
   ② 유전자 검사 권고
   ③ 치료 가이드라인
   ④ 최신 치료 동향
```

### 실측 결과 (MIMIC-IV 폐질환 환자 10명)

| 회의록 §1.4 요구사항 | JSON 매핑 필드 | 통과율 |
|----|----|----|
| ① 근거 기반 질환 제시 | `rag_evidence` + `case_comparison` + `epidemiology_note` | **10/10 (100%)** |
| ② 유전자 검사 권고 | `genetic_test` | 5/10 (50%) |
| ③ 치료 가이드라인 | `treatment_guideline` (`[질환명] 치료법` 형식) | **10/10 (100%)** |
| ④ 최신 치료 동향 | `case_comparison` (PubMed PMID 인용) | 2/10 (20%) |
| 주요 질환 랭킹 설명 | `top1_reasoning` + `differential_note` | **10/10 (100%)** |

### 각 항목 부진 원인 분석

- **② 50%**: 5명은 흡인성 폐렴·COPD처럼 유전자 검사 대상이 아닌 일반 질환 → 임상적으로 정상 동작
- **④ 20%**: PubMed API rate limit(429) 발생으로 일부 환자에서 최신 논문 수집 실패 → NCBI API Key 발급 시 초당 10회로 개선 가능

- `rag/valid/eval_output_structure.py` — 회의록 §1.4 5가지 요구사항 통과율 측정
